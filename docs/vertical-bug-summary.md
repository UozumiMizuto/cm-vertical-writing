# 縦書きエディタ（CodeMirror 6 + CSS Transform）における座標ズレの原因と対策

本ドキュメントは、CodeMirror 6を `transform: rotate(90deg)` で縦書き表示した際に発生した「カーソル位置とクリック位置がズレる（特定の行や文字が選択できない）」という深刻なバグの根本原因と解決策をまとめたものです。

## 1. 根本的な原因の全体像

CodeMirror 6 (CM6) は本来「横書き」を前提として作られており、エディタ内の文字の座標計算（どこをクリックしたか、カーソルをどこに描画するか）は全てDOMの `getBoundingClientRect()` や `Range.prototype.getClientRects()` に依存しています。
今回の実装では、横書きのCM6をCSSの `transform` で90度回転させることで縦書きを表現し、ブラウザの出力する視覚座標（回転後）を逆計算（回転前）してCM6に渡す「座標パッチ」を当てて対応していました。

しかし、以下の3つの要素が複雑に絡み合い、この座標パッチの前提を破壊していました。

---

## 2. 個別の原因と対策

### ① React StrictMode によるパッチの多重適用（プロトタイプ汚染）
- **症状:** 開発環境（ローカルサーバー）でリロードすると、画面がおかしくなったり座標が完全に狂う。
- **原因:** React 18の StrictMode は、コンポーネントのマウント・アンマウントを2回連続で実行します（Mount → Unmount → Mount）。
座標パッチは「元の `getBoundingClientRect` を変数に退避させてから上書きする」という処理を行っていましたが、2回目のMount時に「すでに上書きされた後の関数」をオリジナルとして退避・保存してしまい、座標計算が無限ループや二重変換を起こして壊滅していました。
- **対策:** `installVerticalPatches` 関数に `installed` フラグ（冪等性の担保）を追加し、何度呼ばれても初回起動時しかオリジナル関数を退避・上書きしないように修正しました。

### ② `Decoration.mark` などの DOM 分断（これが最大の原因）
- **症状:** 特定の拡張機能（文字数エミュレータ、`verticalTextPlugin` など）を入れると、その行だけ（あるいは全行）のクリック位置がメチャクチャになる。
- **原因:** CM6の `Decoration.mark` やインラインの `Widget` を使うと、テキストノードの途中に `<span>` などの新しいDOMノードが挿入されます。
ブラウザは通常、行の座標を計算する際に1つのテキストとしての枠を見ますが、途中に装飾用の `<span>`などが挟まる（あるいは `<wbr>` などの改行ヒントが挟まる）と、**ブラウザが返す `Range` (テキストの矩形) の計算結果が複数に分割されたり、高さや幅の算出がズレたりします。**
これをCSSで無理やり90度回転させているため、分割された細かい矩形の逆計算（アフィン変換）が狂い、CM6が「ユーザーがクリックした座標と文字の対応」を完全に見失っていました。
  - **犯人A:** `verticalTextPlugin`（全日本語文字を `<span>` + `<wbr>` で囲っていたため全行壊滅）
  - **犯人B:** 原稿用紙エミュレーター（文字数上限を超えたテキストに `underline wavy` な波線を引く `mark` デコレーションを入れていたため、その行の途中から座標が壊滅）
- **対策:** ノードを分断する `Decoration.mark` やインラインウィジェットの使用を中止しました。代わりに、行全体を対象としてクラスを付与する **`Decoration.line`（行ベースの装飾）** に書き換えました。（エミュレータの赤い波線の代わりに、行全体の背景色と枠線が変化するようになったのはこのためです）。

### ③ フォントサイズや太さを変える装飾の干渉 (`syntaxHighlighting`)
- **症状:** `# 見出し` などのMarkdown書式の機能を使うと、見出し行だけがクリックできなくなる。
- **原因:** `syntaxHighlighting` (デフォルトハイライター) は、見出しに対して個別の `font-size` や `font-weight` を適用します。これにより、同じエディタ内で行の高さ（横書き時の物理的な高さ、縦書き時の幅）が不均一になります。
これも②と同様、CM6の内部計測において特定の行だけ座標系のスケールが変わってしまい、回転と逆計算のパッチがスケールの変化を追いきれずにズレを起こしていました。
- **対策:** 座標系やテキストサイズに影響を与える `syntaxHighlighting` を一時的に削除し、プレーンなフォントサイズで描画されるようにしました。（今後シンタックスハイライトを入れる場合は、フォントサイズを変えずに「色」だけを変えるカスタムテーマを用意する必要があります）。

## 3. 今後の教訓（縦書きエディタ開発における鉄則）

CSS Transformによる縦書き実装において、**「CodeMirrorのDOM（特に行内 `.cm-line` 内部）の構造を変形・分断するような拡張機能」は絶対に使ってはいけません。**

- **NGな実装**
  - インラインの文字への背景色や下線 (`Decoration.mark`)
  - 行の途中へのウィジェット挿入 (`Decoration.widget(inline)`)
  - 行ごとにフォントサイズを変えるハイライト (`syntaxHighlighting`)
- **OKな実装**
  - 行自体のクラスやスタイル変更 (`Decoration.line`)
  - 完全に行と分離・独立して描画されるブロックウィジェット (`Decoration.widget({block: true})` の一部)
  - ルビ `<ruby>` （これはブラウザ標準のレイアウトエンジン内で矩形が完成するため、例外的に座標パッチと相性が良かった）

上級者向けの解決策として、**CSS Custom Highlight API** を使用することで、DOMを分断せずに文字単位の装飾（色付けのみ）を実現可能です。

---

## 4. 抜本的な解決策：CSS Custom Highlight API の採用

2024年以降のモダンブラウザで利用可能な **CSS Custom Highlight API** を使用すると、`<span>` タグなどを挿入せずに「テキストノードの状態を変えずに」スタイルを適用できます。

### メリット
- **DOM構造が一切変化しない**: `getBoundingClientRect()` の結果がプレーンなテキスト時と同じになるため、縦書き座標パッチとの干渉がゼロになります。
- **文字単位の装飾が可能**: 特定の単語や、文字数制限を超えた部分のみを赤くするなどの制御が可能です。

### 制約
- 「色」「背景色」「下線」などの描画系スタイルのみ適用可能（`font-size` や `font-weight` など、レイアウトに影響するスタイルは不可）。
- CodeMirrorの `Decoration.mark` の代わりに、JavaScriptで `Range` を作成して `CSS.highlights` に登録する独自プラグインの実装が必要です。

### 現在の適用状況
- **原稿用紙エミュレータ**: 縦書き時はこのAPIを使用して文字数制限オーバーを「赤字」で表示。
- **DBキーワード連携**: 縦書き時もキーワードを「オレンジ色＋点線」で表示。
- いずれも横書き時は、従来通りの `Decoration.mark` を使用してパフォーマンスと互換性を優先しています。
